Running one AI coding agent is simple. You give Claude Code or Codex a task, it writes the code, you review the result. But try running three or five at the same time on the same repo and things fall apart quickly.

Agents step on each other's files. Two grab the same task because neither knows the other started. When one fails mid-task, the next one has no idea what happened and starts over. You end up babysitting five terminals, manually merging branches, and resolving conflicts that shouldn't have existed in the first place.

We built Stoneforge to solve this. Today we're publicly launching the project and this blog.

## What Stoneforge does

Stoneforge is an orchestration platform for AI coding agents. You install it, start the server, open a web dashboard, and manage a team of agents from one place.

There are four roles in the system. A **Director** takes your high-level goal and breaks it into tasks with priorities and dependency ordering. **Workers** pick up those tasks automatically via a dispatch daemon, and each worker runs in its own git worktree, completely isolated from the others. When a worker finishes, a **Steward** runs your test suite against the branch. Tests pass? Squash-merge. Tests fail? The steward creates a handoff task with context so the next worker can pick up where the last one left off. The **dispatch daemon** ties it together, watching for unblocked tasks and assigning them to idle workers without any manual intervention.

You describe what you want. Agents build it in parallel.

## More than an agent runner

Stoneforge also collapses the project management stack into one system. Tasks and issues live here (priorities, dependencies, kanban views, plan grouping), alongside versioned documents with full-text and semantic search, channel-based messaging with threading and inbox triage, and merge requests with automated test runs. Agents don't need to context-switch between Linear, Notion, Slack, and GitHub because everything they need is in one place.

## How it compares to Claude Code Agent Teams

Claude Code now has an experimental [agent teams](https://docs.anthropic.com/en/docs/claude-code/agent-teams) feature. The two projects solve similar problems but make very different tradeoffs.

Agent teams stores state in ephemeral file-based task lists. Stoneforge event-sources everything to SQLite + JSONL, so state survives restarts and you get a full audit trail. Agent teams is terminal-only; Stoneforge gives you a web dashboard with live agent output, kanban boards, and metrics. For branch isolation, agent teams tells you to "avoid editing the same file." Stoneforge gives each worker its own git worktree automatically. Merging in agent teams is manual. In Stoneforge, stewards run tests and squash-merge without you touching anything.

Stoneforge also supports multiple providers (Claude Code, OpenCode, OpenAI Codex) and lets you split agents across multiple Claude MAX/Pro plans to get around rate limits. Agent teams is Claude-only.

## Architecture

Stoneforge uses a dual storage model. JSONL files are the source of truth: append-only, git-tracked, human-readable, mergeable across branches. Every state change is recorded as an event. SQLite sits on top as a query cache with indexes, FTS5 full-text search, and materialized views. If the SQLite file gets corrupted or deleted, it rebuilds from the event log.

Workers run in isolated git worktrees at `agent/{worker-name}/{task-id}-{slug}`. Each agent gets its own copy of the repo. No branch conflicts, no file locking. When the work is done, the steward merges it back to the main branch.

## Get started

```bash
# Install the CLI
npm install -g @stoneforge/smithy

# Initialize in your project
cd your-project && sf init

# Start the server and dashboard
sf serve

# Open the dashboard
open http://localhost:3457
```

From there, register a Director, some workers, and a merge steward. Tell the Director your goal and let it run.

## Caveats

Stoneforge is early-stage software under active development.

Running multiple agents in parallel burns through tokens. A typical session eats through several Claude MAX/Pro subscriptions worth per week. This is the cost of running a parallel team, and it's intentional.

Agents run autonomously. There are no approval gates before agents take actions, because approval prompts would bottleneck a parallel pipeline. Agents will read, write, execute, and push code without asking.

We ship weekly. Published packages follow semver with changelogs, but defaults may change and docs sometimes lag behind the code.

If one coding agent handles your workload today, you probably don't need Stoneforge yet. It'll be here when you start running multiple agents and need coordination.
