## When do you actually need this?

Not everyone does. If you're running one Claude Code or Codex instance at a time, coordination isn't a problem. The problems show up when you run three or more agents on the same codebase simultaneously.

You're manually splitting work across terminals. You're keeping a mental model of who's doing what. An agent has undone another agent's work because they were both editing the same file. You've waited 20 minutes for one agent to finish a blocking task before starting the next one. You're copying context between agent sessions because they can't see each other's progress.

If that sounds familiar, you're doing multi-agent coordination by hand. Stoneforge automates it.

## The orchestration loop

Everything starts with a goal. You tell the Director "build an authentication system" or "refactor the payments module to use Stripe." The Director is a persistent agent session that acts as your strategic planner. It doesn't write code. It decomposes your goal into a plan.

A plan is a set of tasks with priorities (1 through 5, where 1 is critical) and dependency relationships. The Director might break "build authentication" into:

1. Design the API schema (priority 1, no dependencies)
2. Implement OAuth endpoints (priority 2, blocked by task 1)
3. Build the login UI (priority 2, blocked by task 1)
4. Write integration tests (priority 3, blocked by tasks 2 and 3)

Plans start in draft status. While a plan is in draft, its tasks are invisible to the dispatch system. This prevents a race condition where the daemon assigns a task before the Director has finished wiring dependency relationships. Once the Director activates the plan, tasks become dispatchable.

The dispatch daemon is a background process that polls every 5 seconds. Each cycle runs five phases in strict order:

1. **Orphan recovery** checks for workers with assigned tasks but no active session (happens after server restarts)
2. **Inbox polling** routes messages to agents and spawns triage sessions for accumulated notifications
3. **Steward trigger polling** fires cron-scheduled or event-triggered steward workflows
4. **Workflow task polling** assigns workflow step tasks to available stewards
5. **Closed-unmerged reconciliation** catches tasks stuck at "closed" without being merged (a 2-minute grace period prevents false positives from in-progress merge sequences)

The worker availability check runs within this cycle too. It finds unblocked tasks with no assignee, matches them to idle workers, and dispatches the highest-priority ready task to the next available worker.

Priority sorting goes beyond the number on the task. The daemon walks upstream dependencies to calculate effective priority. If five other tasks are blocked waiting on task A, task A gets bumped up in the queue even if its explicit priority is lower. The traversal goes up to 10 levels deep and considers the minimum (most urgent) priority across all dependents.

```

```

Workers are ephemeral by default. They spawn, execute one task, commit and push their work, then exit. The daemon picks up the next ready task and assigns it to the next idle worker. The loop repeats until the plan is complete.

## The task lifecycle

A task moves through eight possible statuses. Understanding these helps you read the dashboard and debug stuck plans.

**Open** is the starting state. The task exists and is available for assignment. **In progress** means a worker is actively on it. **Blocked** is automatic: Stoneforge transitions a task to blocked when a dependency is added whose blocker hasn't completed yet, and transitions it back when the blocker closes. The system saves the previous status so it can restore it after unblocking. **Backlog** is for tasks that need triage before they're ready for work. **Deferred** is for tasks deliberately postponed. **Review** means the worker finished and the task is waiting for the merge steward to run tests and merge. **Closed** means done and merged. **Tombstone** is a soft delete.

The dispatch daemon's `ready()` query filters for tasks that are open or in progress, not in the blocked cache, not in a draft or blocked plan, not scheduled for a future date, and not part of an ephemeral workflow. Everything else is invisible to dispatch.

## The dependency graph

Dependencies in Stoneforge do more than link tasks. The system supports several dependency types, and only some participate in blocking.

**Blocking dependencies** prevent work from starting:

The `blocks` type is straightforward. Task A blocks task B, so B can't start until A closes. The `parent-child` type creates containment hierarchies. Tasks belong to plans via parent-child dependencies, and libraries contain documents the same way. Blocking is transitive: if a plan is blocked, all its child tasks are blocked too. The `awaits` type handles external gates. A task might await a timer (wait until a specific date), an approval (wait for specific people to sign off), an external system event, or a webhook callback.

**Non-blocking dependencies** capture relationships without preventing work:

`relates-to` creates bidirectional associations. `references` is a unidirectional citation. `supersedes` chains version history. `duplicates` marks duplicate tasks. `caused-by` links a bug fix to the bug that caused it. `validates` links a test task to what it tests. None of these prevent dispatch.

### The blocked cache

Computing "is this task blocked?" by traversing the dependency graph on every poll cycle would be expensive. Stoneforge maintains a materialized view called the blocked cache. It's a SQLite table storing every currently-blocked element along with what's blocking it and the element's previous status.

When a dependency is added or a task's status changes, the cache invalidates and recomputes. When a blocker closes, the cache removes the blocked entry and triggers a status callback that restores the element's previous status. The invalidation cascades through parent-child hierarchies so transitive blocking stays correct.

The cache supports cycle detection via BFS traversal. Before adding a blocking dependency, you can call `detectCycle()` to check whether the new edge would create a circular dependency. The traversal goes up to 100 levels deep and returns the cycle path if one is found.

### Awaits gates

Gates are the interesting part of the dependency graph. A task can await different types of external conditions:

**Timer gates** block until a specified timestamp. Useful for scheduled releases or cooldown periods. Satisfaction is checked automatically: `now >= waitUntil`.

**Approval gates** require sign-off from specific entities. You configure which approvers are required and how many must approve (defaults to all). Approvals are recorded individually and the gate opens when the threshold is met.

**External gates** and **webhook gates** represent events from outside Stoneforge. A CI/CD pipeline completing, a deployment finishing, a third-party API responding. You mark these satisfied manually via `sf dependency satisfy` or programmatically via the API. Once satisfied, the blocked task unblocks and enters the dispatch queue.

## Worktree isolation

This is the most important architectural decision in Stoneforge. Every worker gets its own [git worktree](https://git-scm.com/docs/git-worktree).

A worktree is a separate working directory linked to the same git repository. Each one can have a different branch checked out at the same time. Worker 1 might be editing `src/auth.ts` on `task/abc-oauth` while Worker 2 edits the same file on `task/def-api-routes`. They never interfere because they're in physically separate directories on disk.

```
your-project/
├── .stoneforge/.worktrees/
│   ├── worker-1/task-abc-oauth/          ← branch: task/abc-oauth
│   │   └── src/auth.ts  (worker 1's copy)
│   ├── worker-2/task-def-api-routes/     ← branch: task/def-api-routes
│   │   └── src/auth.ts  (worker 2's copy)
│   └── worker-3/task-ghi-login-ui/       ← branch: task/ghi-login-ui
│       └── src/components/Login.tsx
└── src/                                   ← main branch (untouched)
    └── auth.ts
```

When a worker pushes, it pushes its own branch. The main branch is never modified directly by a worker. Merging is the steward's job.

Worktree paths are deterministic: `agent/{worker-name}/{task-id}-{slug}`. When a task is handed off to a different worker, the new worker resumes in the same worktree on the same branch, picking up exactly where the last worker left off.

Worktrees are created with `git worktree add --no-checkout` followed by `git switch -c` to avoid triggering pre-checkout hooks or build processes during creation. This matters in monorepos where checkout hooks might run expensive operations.

One gotcha worth knowing: if you're using Turborepo, fresh worktrees require `turbo build --force`. Turbo's shared cache can restore build outputs to the wrong worktree path. Stoneforge handles this internally, but it helps to know if you're debugging build issues.

## Two databases, one source of truth

Stoneforge stores everything twice. Every task, document, message, dependency, and agent action is recorded in both JSONL files and a SQLite database. These aren't redundant copies. They serve different purposes.

**JSONL is the source of truth.** Every state change is appended as an immutable record to `.stoneforge/sync/elements.jsonl`. These files are text, line-based, and designed to be committed to git. You can diff them, merge them across branches, and read them with any text editor.

```jsonl
{"type":"element","id":"el-abc123","elementType":"task","title":"Implement OAuth","status":"open","priority":1,"createdBy":"director-1","createdAt":"2026-03-02T10:00:00.000Z"}
{"type":"element","id":"el-abc123","elementType":"task","title":"Implement OAuth","status":"in_progress","priority":1,"assignee":"worker-1","updatedAt":"2026-03-02T10:05:00.000Z"}
{"type":"element","id":"el-abc123","elementType":"task","title":"Implement OAuth","status":"closed","priority":1,"assignee":"worker-1","updatedAt":"2026-03-02T11:30:00.000Z"}
```

Dependencies are stored in a separate file, `.stoneforge/sync/dependencies.jsonl`, with the same append-only structure.

**SQLite is the cache.** It provides indexed queries, FTS5 full-text search across documents, the blocked cache for dependency resolution, and sub-millisecond lookups that would be impractical against flat files. But it's entirely disposable. Delete the `.db` file and run `sf sync import`. The system rebuilds from the JSONL records.

Why not just use SQLite? Binary files don't diff. They don't merge across git branches. If the file corrupts (and with multiple agents writing concurrently, that's a real risk), you lose everything.

Why not just use JSONL? Scanning a flat file for a single task by ID is slow once you have thousands of records. Full-text search requires indexes. Dependency graph traversal requires joins.

The dual model gives you append-only durability with query performance.

### Sync conflict resolution

When JSONL files are merged across git branches, conflicts are resolved with a last-write-wins strategy plus several special cases. Content hash comparison (SHA256, excluding timestamps and sync metadata) detects identical changes. Tags always merge as a union, so you never lose tags during sync. Closed status is sticky: if one side closed a task, the closed version wins regardless of timestamp. Fresh tombstones (within a 30-day TTL) win over live elements. For dependencies, removal is authoritative: if one side deleted a dependency, it stays deleted.

The content hash computation strips `metadata._externalSync` fields before hashing. Without this, every push to an external provider would update the sync metadata, change the hash, mark the element dirty, and trigger another push. The stripping breaks the feedback loop.

### The event audit trail

Every mutation records the actor and timestamp. If you need to debug a task in a weird state:

```bash
sf task events el-abc123
```

Worker 1 was assigned at 10:05. It handed off at 10:45 with notes about a CORS issue. Worker 2 picked it up at 10:50. It completed at 11:30. The merge steward squash-merged at 11:32. Tests passed. Every step is attributed and timestamped.

## The merge steward

When a worker finishes a task, it commits, pushes, and marks the task as ready for review. The merge steward picks it up.

The steward runs in a temporary detached worktree. It never touches the main repository's working directory or HEAD. The process:

1. Create a temporary worktree checked out at `origin/main` (or whatever your target branch is)
2. Squash-merge the worker's branch into the temporary worktree
3. If there's a merge conflict, stop and create a fix task for the next available worker
4. If the merge succeeds, run the configured test command
5. Tests pass? Push the merged commit to the target branch and clean up
6. Tests fail? Create a fix task with the failure output attached

The temporary worktree approach means the main repo is never locked during merges. Multiple stewards can process different branches at the same time. An optimistic locking mechanism using merge status transitions (pending → testing → merging → merged) prevents two stewards from claiming the same task. If a steward tries to transition a task that another steward already claimed, it gets a `MergeStatusConflictError` and backs off.

After the push, the steward fast-forwards the local target branch with `git merge --ff-only`. If that fails (someone else pushed in the meantime), it fetches and retries.

You can configure the merge strategy (squash or regular merge), auto-push, branch deletion after merge, and the test command with its timeout. Defaults are squash-merge, auto-push, auto-delete, and `npm test` with a 5-minute timeout.

## The four steward types

The merge steward is the most visible, but Stoneforge runs four types, each with a distinct responsibility.

**Merge stewards** handle branch integration as described above. They watch for tasks entering review status and process them automatically.

**Docs stewards** scan documentation for accuracy. They verify that file paths referenced in docs actually exist, that internal markdown links resolve, that documented CLI commands match the actual command surface, and that TypeScript type documentation matches the source. When they find issues, they create a worktree, apply fixes, commit, and squash-merge back. Six verification categories run in parallel: file paths, internal links, exports, CLI commands, type fields, and API methods.

**Recovery stewards** handle tasks stuck in broken states: assigned but with no active session, closed but never merged, stuck in progress with no recent activity. They diagnose the issue and either restart the work, create a recovery task, or roll back cleanly.

**Custom stewards** are user-defined. You register a steward with `--focus custom`, attach a playbook (inline or as a reference to a workflow template), and configure triggers. The steward executes your playbook when triggered.

### Triggers: cron and events

Any steward can be triggered by a cron schedule, by a system event, or both.

Cron triggers use standard 5-field cron expressions. `0 3 * * *` fires at 3 AM daily. `*/5 * * * *` fires every 5 minutes. The scheduler parses the expression, calculates the next matching time, and sets a `setTimeout` to fire. After execution, it recalculates and reschedules.

Event triggers fire on specific system events. You can attach a condition expression that filters which events match. If a custom steward's trigger is configured as `event: 'task_closed', condition: "task.tags.includes('cleanup-needed')"`, it only fires when a task with that tag closes. Conditions are evaluated in a strict sandbox with safe token validation.

Steward sessions have safety limits. An idle timeout (default 2 minutes) terminates stewards that stop producing output. A max duration timeout (default 30 minutes) prevents runaway sessions. Both are checked every 30 seconds. The scheduler also prevents overlapping executions: if a cron-triggered steward is still running when the next cron fires, the execution is skipped.

## When things go wrong

Agents fail. They hit context window limits. They get stuck. API rate limits kick in. The system handles all of this without losing work.

### Handoffs

When a worker can't finish a task, it hands off. The worker commits and pushes whatever it has, appends a note to the task description explaining where it left off and what remains, unassigns itself, and exits.

The handoff preserves the branch name and worktree path in the task metadata. When the dispatch daemon assigns the task to the next worker, that worker resumes on the same branch in the same directory. It sees all the previous commits. It reads the handoff notes. It picks up where the last worker stopped.

Handoff history accumulates on the task as an array of entries, each recording the session ID, handoff message, branch name, and timestamp. If a task goes through three workers, you can trace what each one did, why each one stopped, and where each one left off.

### Orphan recovery

If the Stoneforge server restarts (a deploy, a crash, dev mode hot-reload), workers mid-task become orphaned. They have assigned tasks but no active session.

On startup, the daemon's first action is orphan recovery. For each orphaned worker, it tries to resume the previous session using the saved provider session ID. If that works, the worker continues where it was. If resume fails, the daemon spawns a fresh session on the same branch and worktree, giving the new session the full task context and handoff notes.

The system tracks resume attempts per worker to prevent infinite retry loops. After three failed attempts, the task is left in review status as a safety valve.

### Rate limiting

When a worker session exits within seconds of starting (the threshold is 10 seconds), that usually means an API rate limit. The dispatch daemon detects this pattern and puts the worker on a one-hour cooldown. During cooldown, the worker is skipped during dispatch. The minimum floor for any rate limit cooldown is 15 minutes, even if the provider signals a shorter reset.

Before spawning triage sessions, the daemon also checks rate limit status. If all configured executables for a steward are rate-limited, the triage spawn is deferred and messages stay unread until limits expire.

## Agent pools and concurrency control

Pools limit how many agents run simultaneously. This is useful for managing API costs, preventing machine resource exhaustion, or splitting agents across different provider plans.

A pool has a `maxSize` (how many agents can run at once) and a list of `agentTypes` that specify which agents count against it. You can filter by role (worker vs steward), worker mode (ephemeral vs persistent), steward focus (merge, docs, recovery, custom), and provider.

Before spawning any agent, the dispatch daemon checks all pools that govern that agent type. The agent must pass every governing pool's capacity check to spawn. If a pool is full, the daemon waits until a slot opens.

Directors are never pool-managed. They're persistent sessions that stay running and don't compete for worker slots.

## Messaging and channels

Agents communicate through a channel-based messaging system. Every agent gets a dedicated direct channel at registration for dispatch notifications, triage messages, and agent-to-agent communication.

Two channel types exist. **Direct channels** are 1:1, created between two entities. Their names are deterministic: sorted entity IDs joined with a colon, so the same pair always resolves to the same channel. Membership is immutable. **Group channels** support multiple members with configurable visibility (public or private) and join policy (open, invite-only, or request-based). Designated moderators control membership.

Messages are immutable after creation. They can't be edited or deleted. Any attempt throws a `MessageImmutableError`. This gives you a reliable audit trail of all agent communication.

Message content is stored as a separate Document element, referenced via `contentRef`. This keeps message metadata lean, supports large content without bloating the message table, and means message content is searchable through the same full-text search that indexes documents. Attachments are an array of Document IDs, up to 100 per message.

Threading works through a `threadId` field. Root messages have `threadId: null`. Replies point to the root message's ID. You can retrieve entire threads with `sf message thread <msg-id>`.

### Inbox and message routing

The inbox service tracks unread messages per agent with three source types: `direct` (from a DM channel), `mention` (the agent was @mentioned), and `thread_reply` (someone replied to a thread the agent started). Items progress through three statuses: unread → read → archived.

The dispatch daemon routes messages differently depending on the agent's role and session state:

Dispatch messages (task assignments) are marked as read immediately. The actual session spawn is handled separately by worker availability polling.

For ephemeral workers that are busy on a task, non-dispatch messages are left unread. Interrupting a focused task session would be counterproductive. When the worker finishes and becomes idle, the daemon spawns a **triage session**: a lightweight, read-only session where the agent processes accumulated messages. The daemon groups unread messages by channel, picks one channel, creates a temporary detached worktree, and spawns the agent with a triage prompt listing all pending messages. The agent can respond, create tasks, escalate to the Director, or acknowledge. Only one channel's messages are processed per triage session. Remaining channels are handled in subsequent poll cycles.

For persistent workers and Directors with active sessions, messages are forwarded directly as user input. A duplicate guard using an in-flight tracking set prevents the same message from being forwarded twice if the polling cycle runs faster than the forwarding completes.

### Versioning

Every update creates a new version. The previous version is snapshotted to a `document_versions` table before the update is applied. You can view the full history with `sf document history <id>`, view a specific version with `sf document show <id> --doc-version 2`, or roll back by creating a new version with old content. Rollback doesn't erase history. It creates version N+1 with the old content.

### Libraries

Documents are organized into libraries via parent-child dependencies. A library can contain documents and sub-libraries, forming a tree. Documents can belong to multiple libraries. Libraries support hierarchical nesting (with cycle detection to prevent circular containment), and the system can resolve a document's full library path: `documentation/api-reference/authentication`.

### Search

Two search modes are available.

**Full-text search** uses SQLite's FTS5 engine with Porter stemming and Unicode support. Queries are sanitized by splitting into tokens and wrapping each in double quotes to prevent FTS5 operator interpretation. Results are ranked by BM25 relevance score. An adaptive top-K filter uses elbow detection on score gaps to find the natural quality cutoff.

**Semantic search** is optional and uses vector embeddings stored as Float32Array blobs. When both modes are available, a hybrid search combines rankings using Reciprocal Rank Fusion (RRF) with a smoothing constant of k=60.

### The descriptionRef pattern

Tasks don't store their descriptions inline. They reference a Document via `descriptionRef`. This lets descriptions be rich (markdown, arbitrary length), versioned independently from the task, and searchable through full-text search. When a worker hands off, the handoff notes are appended to this description Document so the next worker gets the accumulated context.

Messages use the same pattern via `contentRef`, and libraries can have descriptions via `descriptionRef`. The pattern keeps metadata tables lean while supporting arbitrarily complex content.

## External sync

Stoneforge can sync data bidirectionally with external systems. Four providers are supported: GitHub (issues and PRs), Linear (issues), Notion (documents as pages), and folder (documents as markdown files on disk).

Each sync-enabled element stores its external link in `metadata._externalSync`, including the provider, project, external ID, and content hashes from the last push and pull. Change detection compares current content hashes against stored hashes to determine what needs syncing.

The sync engine supports push, pull, and bidirectional modes. Conflict resolution is configurable: last-write-wins (default), local-wins, remote-wins, or manual (tags the element with `external-sync-conflict` for human resolution).

Not everything syncs. System-category documents (`task-description` and `message-content`) are excluded automatically. Documents without titles are excluded to prevent untitled stubs from overwriting real content in external systems.

## Workflows and playbooks

For multi-step processes that need to execute in order with durable state, Stoneforge provides workflows and playbooks.

A **playbook** is a reusable template. It defines a sequence of steps (task steps or function steps), template variables with types and defaults, and optional conditions that include or exclude steps based on variable values. Playbooks support inheritance: a playbook can extend another, combining their steps and variables.

A **workflow** is a runtime instance of a playbook. When you instantiate one, Stoneforge resolves inheritance, validates and fills variables, evaluates step conditions, creates Task elements for each included step, wires blocking dependencies between steps based on `dependsOn` declarations, and creates parent-child dependencies linking all steps to the workflow.

Workflows can be ephemeral (for one-off or test runs, not persisted to JSONL) or durable (fully event-sourced). The status lifecycle is: pending → running → completed or failed or cancelled.

**Function steps** can execute TypeScript, Python, or shell commands with configurable timeouts. Task steps are regular Stoneforge tasks dispatched to workers. Because dependencies between steps are expressed as blocking dependencies, the normal dispatch system handles ordering: step 2 won't start until step 1 closes.

If a step fails, the workflow's state is preserved. You can fix the issue, reopen the failed task, and the workflow resumes from that step. This is the "durable state" property: the workflow's progress is the aggregate status of its child tasks, not a separate counter that can drift out of sync.

## Custom prompts

Each agent role has a built-in system prompt that defines its behavior. The Director prompt explains how to decompose goals into tasks, set priorities, and manage dependencies. The worker prompt explains how to execute tasks, commit, push, and hand off. Steward prompts explain the specific focus area (merge, docs, recovery).

You can override any of these per-project by placing `.md` files in `.stoneforge/prompts/`. The file names match the built-in prompt names: `director.md`, `worker.md`, `persistent-worker.md`, `steward-base.md`, `steward-merge.md`, `steward-docs.md`, `steward-recovery.md`, and `message-triage.md`.

Overrides take full precedence. If you provide a `worker.md`, the built-in worker prompt is replaced entirely. For stewards, you can override the base prompt and focus prompt independently. The system tracks which prompts are built-in and which are project overrides, so you can debug unexpected behavior.

Prompts support template variables. `{{baseBranch}}` is replaced with the detected target branch at render time. This lets prompts reference the correct branch for merge operations without hardcoding `main` or `master`.

## Providers and multi-plan scaling

Stoneforge abstracts the underlying agent CLI through a provider system. Three providers are built in:

**Claude Code** is the default. Headless sessions use the `@anthropic-ai/claude-agent-sdk` for structured message streaming. Interactive sessions spawn a Claude Code CLI process via node-pty for terminal-based output.

**OpenCode** connects through an HTTP server using `@opencode-ai/sdk` for headless sessions and the OpenCode CLI for interactive sessions.

**OpenAI Codex** uses a JSON-RPC protocol over stdio for headless sessions and the Codex CLI for interactive sessions.

Each provider handles its own authentication. You don't configure API keys in Stoneforge. Claude Code uses `~/.claude/`, OpenCode uses its own config directory, and Codex uses whatever OpenAI authentication is configured. Stoneforge doesn't need to know or store credentials.

The provider is set per-agent at registration (`--provider opencode`) or per-session at start. You can mix providers within the same orchestration: a Director running on Claude Code, workers on Codex, stewards on OpenCode. The dispatch system doesn't care which provider an agent uses.

### Multi-plan scaling

If you're running 10+ agents, you'll hit rate limits on a single Claude MAX or Pro plan. Stoneforge supports splitting agents across multiple plans through the executable path system.

Each agent can specify a custom `executablePath` that points to a wrapper script. The wrapper sets `CLAUDE_CONFIG_DIR` to a plan-specific directory and calls the real Claude CLI. You create one wrapper per plan, authenticate each one separately, and register different workers with different executable paths.

```bash
#!/bin/bash
# ~/.stoneforge/claude-plan-2.sh
export CLAUDE_CONFIG_DIR=~/.claude-plan-2
exec /usr/local/bin/claude "$@"
```

Register workers with `--executable-path ~/.stoneforge/claude-plan-2.sh`. The dispatch daemon routes tasks to workers regardless of which plan they're on. If one plan's workers are rate-limited, other plans' workers keep going.

The sidebar groups pages by function. **Overview** has Activity (live agent output), Inbox (messages needing attention), and Editor (in-browser code editor with Monaco and LSP support). **Work** has Tasks (list and kanban views), Merge Requests (review queue), Plans (grouped tasks with progress tracking), and Workflows (durable step sequences). **Orchestration** has Agents (register, start, stop, pools, topology graph) and Workspaces (tmux-like terminal multiplexer with saved layouts). **Collaborate** has Messages (channels) and Documents (knowledge base with search). **Analytics** has Metrics (throughput, efficiency, queue health).

The dashboard connects to the server via WebSocket. The event broadcaster polls the database every 500ms and pushes updates to connected clients. Dashboard updates have up to half a second of latency, but for a system where tasks take minutes, the delay is imperceptible.

## The identity system

Stoneforge supports three identity modes. **Soft mode** (the default) is name-based with no cryptographic verification. You set an actor name and go. **Cryptographic mode** requires Ed25519 signatures on every mutation. Entities register with public keys, and each request includes a signature, timestamp, and request hash. The server verifies the signature against the stored public key and checks that the timestamp is within a tolerance window (default 5 minutes). **Hybrid mode** accepts both signed and unsigned requests.

For most single-developer setups, soft mode is fine. Cryptographic mode is there for environments where you need to prove which entity made which change, or prevent agents from impersonating each other.

## Setting it up

```bash
npm install -g @stoneforge/smithy
cd your-project
sf init
sf serve
```

Open `http://localhost:3457`. Register a Director from the Agents page. Start the Director, give it your goal through the Director Panel. Register a few ephemeral workers and a merge steward. Start the dispatch daemon with `sf daemon start`.

The Director creates a plan. The daemon assigns tasks. Workers execute in isolated worktrees. Stewards merge the results. You watch from the dashboard.

## Tradeoffs

**Polling, not pub/sub.** The dispatch daemon polls every 5 seconds instead of using a message queue or event stream. This adds up to 5 seconds of latency between a task becoming ready and a worker picking it up. Polling is simpler to implement, debug, and operate. For a system where tasks take minutes to hours, 5 seconds of dispatch latency doesn't matter. The interval is configurable down to 1 second.

**SQLite, not Postgres.** The entire data layer runs on SQLite. Everything is local, there's no database server to manage, and the system works offline. The tradeoff is that you can't run multiple Stoneforge instances against the same data. For a tool designed to run on a single developer machine, that's acceptable.

**Worktrees, not containers.** We considered running each worker in a Docker container for stronger isolation. Worktrees are faster to create (milliseconds vs. seconds), share the same `node_modules` and build cache, and don't require Docker. The isolation is at the git level, which is sufficient for the conflicts we're preventing.

**No approval gates.** Stoneforge spawns agents with permissions bypassed. No confirmation dialogs. If you're running five agents in parallel and each one needs approval for every file write, you've built a sequential system with extra steps. The mitigation is code review at the merge steward level, not permission prompts at the agent level.

**Immutable messages, mutable tasks.** Messages can't be edited or deleted. Tasks can be updated freely. Messages are an audit trail of communication. Tasks are living artifacts whose status, priority, and description change throughout their lifecycle.

**Event sourcing overhead.** Recording every mutation as an immutable record means the events table grows indefinitely. No archival or pruning exists yet. Complete auditability at the cost of ever-growing storage. For a single-developer tool, storage growth is slow enough that it's not a practical concern yet.
